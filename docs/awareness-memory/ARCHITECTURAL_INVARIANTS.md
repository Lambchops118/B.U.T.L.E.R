# Architectural Invariants

These constraints apply throughout the subsystem. Phase documents cite the identifiers rather than restating every rule.

| ID | Rule and rationale | Prohibited interpretation | Affected phases |
|---|---|---|---|
| INV-01 | Model current state, immutable history, numeric telemetry, working/situational state, and validated long-term memory separately so each supports its own semantics. | Treating the newest event, transcript, or vector hit as authoritative current state; treating telemetry as semantic memory. | 1-8 |
| INV-02 | Deterministic code owns ingestion, validation, authorization, deduplication, ordering, freshness, state, rules, alerts, retries, retention, and action validation so continuous awareness does not depend on inference. | Using the LLM as event loop, database, safety controller, alert detector, retry worker, or arbitrary executor. | 1-8 |
| INV-03 | Exact current, numeric, and temporal questions use bounded structured state, event, or time-series retrieval; semantic search is reserved for similarity. | Defaulting exact questions to vector search or asking the model to infer facts from prose. | 3, 5, 6 |
| INV-04 | Assume distributed messages may be duplicated, delayed, reordered, interrupted, or missing; important operations are idempotent and recoverable. | Assuming broker QoS or network order provides end-to-end exactly-once behavior. | 1-8 |
| INV-05 | The central local database is authoritative for event, state, alert, memory, action, and delivery records; MQTT is transport. | Treating retained MQTT messages or a transient cache as authoritative without provenance/freshness validation. | 1-8 |
| INV-06 | Preserve provenance, confidence, source health, clock quality, freshness, uncertainty, observation/receipt/processing times, temporal validity, and evidence links. | Flattening timestamps or presenting stale, inferred, conflicting, or untrusted information as certain and current. | 1-8 |
| INV-07 | Keep the solution local-first and require no cloud database, vector store, embedding, or inference service. | Sending stored data externally except through an existing explicitly configured integration for its intended purpose. | 0-8 |
| INV-08 | Alerts and deterministic safety-related backend behavior continue without Ollama, using deterministic fallback notification wording. | Delaying a critical alert for model-generated prose or claiming model-dependent work completed while unavailable. | 4-8 |
| INV-09 | Immediate electrical and mechanical interlocks remain in firmware or hardware. | Treating backend rules, network commands, or the LLM as a substitute for local shutdown. | 0, 2, 4, 7, 8 |
| INV-10 | Preserve existing functionality and integrate additively through repository conventions and adapters. | Rewriting unrelated working systems, changing established MQTT topics without migration, or creating fake integrations. | 0-8 |
| INV-11 | Prefer a modular monolith unless discovery proves the repository already has another suitable established architecture. | Creating a microservice fleet or placeholder module tree merely to mirror the specification. | 0, 1-8 |
| INV-12 | Model context is selected, bounded, relevant, temporally explicit, and audited; critical alerts survive truncation. | Dumping raw streams, whole registries/databases, unbounded tool results, or all transcripts into context. | 5, 6, 8 |
| INV-13 | Physical actions use registered definitions, strict parameters, permissions, confirmations, safety checks, idempotency, timeouts, acknowledgements, state confirmation where possible, and transition audit. | Arbitrary MQTT publishing, shell/file execution, bypassed confirmation, invented actions, or treating silence as success. | 1, 4, 7, 8 |
| INV-14 | Failures and degradation are reported truthfully with persisted evidence. | Claiming writes, notifications, embeddings, commands, or backups succeeded without evidence. | 1-8 |
| INV-15 | Use a transactional outbox for durable downstream work; workers are bounded, idempotent, retryable, and observable. | Network/LLM work inside event transactions, unbounded queues/retries, or claims of true end-to-end exactly once. | 1, 4, 6-8 |
| INV-16 | Retention is configurable, dry-runnable, bounded, resumable, aggregate-before-delete, and protects unresolved alert evidence and memory provenance. | Broad deletion without preview/tests or deleting raw evidence before required aggregates/provenance are safe. | 1, 3, 6, 8 |
| INV-17 | Schemas are strict and versioned; inputs, payload sizes, time ranges, batches, queues, retries, result sets, and context have enforced limits. | Dynamic unversioned tables, arbitrary input, unlimited history/tool output, or unbounded in-memory buffering. | 1-8 |
| INV-18 | Existing suitable technology takes precedence; documented stack choices remain defaults and substitutions must preserve transactions, durability, temporal/semantic retrieval, migrations, backup, and concurrency safety. | Converting defaults into unconditional mandates or silently downgrading to memory dictionaries/unversioned flat files. | 0, 1, 6, 8 |
| INV-19 | Every phase keeps the repository runnable, runs relevant tests, updates documentation/status/handoff, reports honestly, and stops at its boundary. | Beginning future-phase work, weakening tests, or claiming subsystem completion from one phase. | 0-8 |
| INV-20 | Phase 0 performs repository/deployment discovery only and stops for owner review before runtime scaffolding unless explicitly waived. | Inferring repository-dependent decisions or beginning Phase 1 implementation during discovery. | 0, entry to 1 |
