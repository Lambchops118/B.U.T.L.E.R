# Session Handoff — 2026-07-18 — Spoken Tool-Call JSON (Leaked Tool Calls)

```text
Session goal: Investigate and fix the system speaking raw tool-call JSON aloud on tool queries ("what's the temperature", "are there any calls recently").
Current phase: Phases 0-8 complete; owner-authorized post-completion bounded runtime fix.
Root cause (two layers):
  Deep root: the migration from OpenAI's Responses API (tool calls are structurally separate from text and cannot leak) to a local model that emits tool calls as TEXT the server must parse. A local Qwen3 occasionally prints the call as plain content -- a bare {"name": ..., "arguments": ...} object or <tool_call>...</tool_call> -- instead of returning it structured. That content is streamed to TTS and spoken. This class of failure was impossible on the hosted API, which is why it "never happened before".
  Amplifier: in-context mimicry (same mechanism as the time/pleasantry bugs). run_command_stream stores the spoken response_text via _record_memory_turn, so a single leaked turn is persisted and replayed; the model then imitates the JSON pattern and leaks again -- a self-perpetuating loop. Proven: with clean history the model NEVER leaked across many runs (minimal prompt, real 9.9K prompt, 2-round loop, 34 complex tools); with history containing a prior assistant turn that "spoke" tool JSON, it leaked intermittently (~50%). Ollama /v1 parsing itself is fine on clean history; the template is the correct Qwen3 one.
Fix implemented (owner chose option 1: suppress + sanitize + recover, wanting the root addressed, not a mute):
  In run_command_stream, a per-turn streaming guard makes an early speak-vs-suppress decision from the first non-whitespace content: natural text streams immediately (no added TTS latency); a turn whose content begins with '{' or '<tool_call' is held back. After the turn, _extract_leaked_tool_calls parses the markup (both bare JSON objects, incl. multiple, and <tool_call> wrappers) and those calls are executed exactly like structured ones, so the tool runs and the model produces a real spoken answer next round. The raw markup is never yielded to TTS and never added to full_text_parts, so it is neither spoken nor stored -- which also breaks the mimicry loop. A false alarm (a natural reply that merely starts with '{') is detected (no parseable call) and spoken normally. Gated by TALOS_RECOVER_LEAKED_TOOL_CALLS (default on). The prior task's history reset already cleared the active amplifier.
```
